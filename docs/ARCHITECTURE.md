# Kora — Architecture

## Overview

Kora is a **config-driven application engine** written in Go. You define your application — its data model, rules, permissions, and workflows — in YAML config files. The engine reads those files and provides a fully functional backend: database schema, REST API, admin UI, and background jobs.

The engine is **generic and permanent**. Applications are **configurations**. Deploying a new feature is a config change, not a code deployment.

### What Kora Is Not

- **Not a code generator.** It does not produce Go source files from config.
- **Includes a visual builder** — the Administrator tab in the workspace lets you create doctypes, permissions, and workflows from the browser with live YAML preview.
- **Not prescriptive** about how configs are created — they can be written by hand, generated by an AI agent, produced by a visual editor, or emitted by another tool.

## Core Concepts

### Config as Source of Truth

Every structural aspect of a Kora application is expressed in config:

| Config File | Defines |
|---|---|
| `doctypes/*.yaml` | Data entities, fields, constraints |
| `roles.yaml` | User roles |
| `permissions.yaml` | Who can do what (RBAC) |
| `*_workflow.yaml` | Document lifecycle (states + transitions) |
| `scheduler.yaml` | Background jobs |

Config is stored in the **database** (`_kora_doctype`, `_kora_field`, etc.), not on the filesystem. YAML files are a one-shot import mechanism. After import, the database is the source of truth.

### DocType

A DocType is the core config unit. It defines a data entity. Every DocType maps to a database table prefixed with `tab`.

Example: `Customer` → database table `tabCustomer`.

### Document

A document is a single row in a DocType's table. All documents are represented generically as `map[string]any` at runtime. The engine is type-agnostic — it validates values against the field's declared type and constraints.

### Registry

The Registry is an in-memory data structure built from the active config version at startup. It holds:

- All DocType definitions (fields, types, constraints)
- The Permission Matrix (role → doctype → allowed operations)
- The Workflow Map (doctype → state machine)

It is **read-only** during normal operation. Updates go through the config versioning system.

### Multi-Tenancy (Sites)

A single Kora instance can serve multiple sites. Each site has:

- Its **own database** (complete isolation)
- Its **own config** (different DocTypes, permissions, workflows)
- One or more **domains** (Host header routing)

**Three access methods, all automatic:**

| Method | Example | Use case |
|---|---|---|
| **Path-based** | `localhost:8000/s/airtime/workspace` | Local dev, no DNS needed |
| **Host-based** | `airtime.myapp.com/workspace` | Production with DNS |
| **Default fallback** | `localhost:8000/workspace` | Single site or dev |

Path-based access sets a `kora_site` cookie so API calls know which site context to use. Host-based access reads the `Host` header. Both coexist — path-based takes priority.

## Request Flow

### Startup Sequence

```
1. kora serve
2. Load config from env vars (CommonConfigFromEnv)
3. Connect to platform DB (MySQL or remote LibSQL via DB_DSN)
4. Discover sites from DB: SELECT DISTINCT site FROM _kora_config_version
5. For each site:
   a. Reconstruct site config from platform defaults + persisted domains
   b. Connect to database
   c. Bootstrap _kora_* system tables (dialect-aware DDL)
   d. Load config from database → Registry
   e. Run schema migration (dialect-aware)
5. Build SiteRouter (domain → site map)
6. Register middleware stack
7. Register API routes
8. Start scheduler
9. Listen (HTTP or HTTPS with autocert)
```

### Authenticated API Request

```
1. TLS handshake (autocert: Let's Encrypt)
2. Recovery middleware (panic → 500)
3. RequestID middleware (X-Request-Id)
4. Security headers (CSP, HSTS, XFO, XSS)
5. CORS headers (Access-Control-*)
6. SiteRouter: Host header OR kora_site cookie OR /s/:site/ path → site context
7. Rate limiter: token bucket check
8. Auth middleware: validates `kora_sid` cookie or `Authorization: Bearer` header → session in **site's DB** → user + roles. No `c.Next()` call — validation runs before CSRF check.
9. CSRF check (POST/PUT/DELETE only): double-submit cookie pattern with constant-time comparison. Runs BEFORE the handler (SiteGuard ordering fix).
10. Workflow actions: available transitions evaluated against the real document with server-enforced conditions.
11. Constraint validation: field + document rules
12. ORM operation: INSERT/UPDATE/DELETE/SELECT
13. Response envelope: {data, meta}
```

**Site-aware auth:** Login, session creation, session validation, and logout all use the site database from the request context (`site_db`), NOT a hardcoded database. This means:

- Login at `/s/airtime/workspace` → creates session in `airtime_db`
- Login at `/s/fieldwork/workspace` → creates session in `fieldwork_db`
- Sessions are site-scoped — a session from one site doesn't work on another
- The `kora_site` cookie tells the SiteRouter which DB to use for all API calls
```

## Component Details

### Expression Engine

Uses `expr-lang/expr` for safe, sandboxed expression evaluation. Used for workflow conditions, constraint conditions, and computed fields.

**Workflow/constraint expressions** have access to:

| Variable | Description |
|---|---|
| `doc` | Current document (all field values) |
| `user` | Current user (name, roles) |
| `today()` | Today's date |
| `now()` | Current datetime |
| `len(x)` | Length of string or array |
| `has_role('Role')` | Check if user has a specific role |

**Computed field expressions** support:

| Function | Description |
|---|---|
| Arithmetic | `+`, `-`, `*`, `/` |
| `SUM(table.column)` | Sum a child table column |
| `COUNT(table)` | Count child table rows |
| `ROUND(expr, N)` | Round to N decimal places |
| `DATEDIFF(a, b)` | Days between two dates |

Values are normalized from DB strings to float64 so expressions like `doc.total > 0` work correctly on persisted data. Document fields from MySQL (`[]byte`) are converted to numeric types during evaluation.

### Permission System

Permissions are defined per DocType, per Role. The engine enforces them at the API layer.

| Operation | Meaning |
|---|---|
| `read` | Can view documents |
| `write` | Can update documents |
| `create` | Can create new documents |
| `delete` | Can delete documents |
| `submit` | Can execute workflow transitions |
| `if_owner` | Above operations apply only to own documents |

Administrator role bypasses all checks. Other roles **default to denied** — permissions must be explicitly granted.

**New Doctype Default:** When a doctype is created, only the Administrator role gets permissions automatically. All other roles must be explicitly granted access via the Permissions panel (`/workspace/admin/permissions`). This follows the principle "explicit is better than implicit" and prevents accidental data exposure.

**Frontend Integration:** The SPA reads permission data from `GET /api/system/doctype/:name` (the `permissions` field in the schema response) and gates UI elements:
- No `create` → "New" button disabled (greyed out) with tooltip
- No `write` → Save button disabled, form fields disabled, "(read-only)" badge

Permissions are also reloaded into the in-memory registry after a new doctype is created, so changes take effect immediately without a server restart.

### System Console

A separate `/console` endpoint serves a React SPA for system administration — site creation, health monitoring, and site management. Config management (doctypes, permissions, workflows, versions, users, secrets) is done via the **Administrator tab** in the workspace SPA. The console uses **separate authentication** from site workspaces:

- **Credentials**: `system_credentials.yaml` — email + bcrypt (cost 12) password, hashed at startup
- **Auth methods**: HTTP Basic auth (`Authorization: Basic base64(email:password)`) or `kora_console_sid` session cookie
- **Session store**: In-memory with cryptographically random 64-char session IDs, 24h expiry, background cleanup
- **Middleware**: `SystemGuard` validates sessions server-side — no cookie forgery possible

### Workflow Engine

Documents can have lifecycles defined as state machines.

```
Draft → Submitted → Approved → In Progress → Completed
               ↘ Rejected → Draft
```

Each transition can have:
- **Allowed roles** — who can perform it
- **Conditions** — expression that must be true
- **Required fields** — fields that must be set
- **Notifications** — emails fired on state change

### Schema Migration

The engine compares the Registry against the live database schema (`INFORMATION_SCHEMA`) and applies additive DDL automatically:

| Detected | Action |
|---|---|
| DocType has no table | `CREATE TABLE` |
| Field added | `ALTER TABLE ADD COLUMN` |
| `search_index: true` | `CREATE INDEX` |
| `unique: true` | `CREATE UNIQUE INDEX` |
| Field removed | Warning logged; column orphaned (not dropped) |

**Safety Tiers:** Before activation, changes are classified by `schema.AnalyzeImpact()`:

| Tier | Change | Action |
|------|--------|--------|
| Safe | New doctype, nullable field, index | Auto-apply |
| Warning | Required field no default, orphan column | Show impact, confirm |
| Blocked | Type change, rename without migration | Refuse without explicit fix |

### Computed Fields

Fields with a `computed` expression are auto-calculated and **persisted server-side** on Insert and Save. The frontend also evaluates them client-side for instant feedback when editing.

```yaml
- fieldname: line_total
  computed: "quantity * unit_price"

- fieldname: subtotal
  computed: "SUM(items.line_total)"

- fieldname: total
  computed: "ROUND(subtotal - discount, 2)"

- fieldname: days_left
  computed: "DATEDIFF(today(), end_date)"
```

**Server-side evaluation:** The ORM's `Insert` and `Save` methods call `ComputeFields()` which:
1. Evaluates child table computed fields first (e.g., `line_total`)
2. Evaluates parent fields with `SUM`/`COUNT`/`DATEDIFF` next (e.g., `subtotal`)
3. Evaluates dependent `ROUND` fields last (e.g., `total`)
4. Persists all computed values via a follow-up UPDATE

**Supported expressions:** arithmetic (`+`, `-`, `*`, `/`), `SUM(table.column)`, `COUNT(table)`, `ROUND(expr, N)`, `DATEDIFF(a, b)`, `today()`.

### Linked Field Auto-Population

The `linked_field` property auto-fills a field from a linked document when a Link field changes:

```yaml
- fieldname: product
  fieldtype: Link
  options: Product

- fieldname: unit_price
  linked_field: "product.selling_price"  # fetch from Product
```

When the user selects a Product, the frontend fetches `GET /api/resource/Product/{name}`, reads `selling_price`, and sets `unit_price`. `computed` expressions depending on `unit_price` cascade automatically.

### Background Jobs

Scheduled via cron expressions in `scheduler.yaml`. Two built-in job types:

| Type | Description |
|---|---|
| `doctype_alert` | Query a DocType; notify users if matches found |
| `email_report` | Query a DocType; send results as formatted email |

### AI Chat System

The AI Chat at `POST /api/chat` auto-generates OpenAI-compatible function definitions from the doctype registry, enabling natural language CRUD operations. Supports OpenAI, DeepSeek, and Anthropic providers via `/chat/completions`.

**Tool Generation:** For each non-child DocType, four tools are generated per doctype: `_find`, `_list`, `_get`, `_create`. System tools (`list_doctypes`, `validate_doctype_yaml`, `create_doctype_draft`) enable AI-assisted doctype creation — always as Draft, never activated by AI.

**Multi-Round Loop:** `finish_reason`-driven (same pattern as Anthropic SDK, OpenAI SDK, LangChain):
- `finish_reason: "stop"` → return final response
- `finish_reason: "tool_calls"` → execute tools, feed results back, loop (tools always present)
- `finish_reason: "length"` → return truncated response
- `finish_reason: "content_filter"` → return policy message

**Safety Nets (all configurable via `AIConfig`):**
- Max rounds fallback cap + stall detection (3× identical call → nudge)
- Tool error circuit breaker (5 errors → force respond)
- Context compaction at 80% token budget
- Tool result size cap (4KB head+tail)
- Textified tool call detection (DeepSeek V4 `<｜｜DSML｜｜tool_calls>` pattern)
- Narrate-then-act false finish detection (GPT-4o)
- HTTP retry with exponential backoff on 429/503

**AI Audit Trail:** AI-created records use `owner = <user>` and `modified_by = "ai-assistant"`. Query `WHERE modified_by = 'ai-assistant'` for audit.

**AI Configuration:** Per-model defaults in `api/ai_config.go` (GPT-4o: 120K tokens, Claude: 190K, DeepSeek: 120K). Site-level overrides via `_kora_secret` keys prefixed `ai.`.

**Frontend:** Floating chat widget (`ui/src/components/chat/ChatWidget.tsx`) embedded in root layout, available on every page.

**MCP Server:** Separate stdio-based MCP server (`mcp/server.go`) for Claude Desktop integration — auto-generates 5 tools per doctype.

### Analytics Engine

The analytics engine provides real-time metrics and insights auto-generated from DocType metadata. It's an **opt-in** system — enabled by setting `KORA_ANALYTICS=true` and `KORA_MYSQL_BUS_HOST` (the bus is a separate MySQL server to avoid interfering with operational workloads).

**Architecture:** EventBus (in-memory channel + WAL drain to MySQL) → Worker (aggregation) → Daily/Monthly rollup tables → Query Engine (SQL-based) → REST API → Frontend Insights tab + Analytics admin page.

```
Document Write (ORM)
       │
       ▼
   EventBus (in-memory channel, async)
       │
       ▼
   Worker pool (UPSERT daily rollup, track workflow transitions)
       │
       ▼
   _kora_analytics_daily / _kora_analytics_monthly / _kora_analytics_transition
       │
       ▼
   Query Engine (SQL aggregation: COUNT, SUM, AVG, MIN, MAX, funnel, duration)
       │
       ▼
   GET /api/analytics/metrics → POST /api/analytics/metrics/:name/query → Frontend charts
```

**Auto-generated Metrics:** Every DocType gets `total_<doctype>` (COUNT) and one metric per Select/Link field (grouped count). Submittable doctypes get additional funnel metrics per workflow state and `avg_duration_<doctype>` for completed workflows.

**Query Engine** supports:
- Time series (daily/weekly/monthly granularity, date range filtering)
- Group-by dimensions (Select/Link fields)
- Funnel analysis (workflow state transitions with fall-through rates)
- Duration tracking (time-in-state for workflow documents)
- Custom metrics (user-defined via POST /api/analytics/metrics)

**Retention & Cleanup:** A background job purges events older than configured retention (default 90 days). WAL table is drained into the daily rollup every 5 seconds.

**CLI:** `kora analytics backfill --site X` reprocesses all historical events from the WAL.

### Admin UI (Workspace)

React SPA served at `/workspace`. Built with Vite, embedded in the Go binary via `go:embed`. All views are **config-driven** — the UI has no knowledge of specific doctypes. It reads schemas from `/api/system/doctype/:name` and renders accordingly.

**Tech stack:** React 19 + TanStack Router/Query/Table/Form + shadcn/ui + Tailwind CSS v4 + Zustand.

**Auto-generated views:**
- **List view** — columns from `in_list_view: true` fields. Field type determines cell renderer (Select→badge, Link→clickable, Date→formatted, Check→icon). Sortable by column click. Offset pagination.
- **Form view** — every field type maps to a widget (Data→Input, Text→Textarea, Select→Dropdown, Link→searchable autocomplete, Table→inline child table grid, Section Break→collapsible section, etc.). `depends_on` visibility. Zod validation from constraints.
- **Workflow bar** — current state badge (color from workflow config). Action buttons filtered by user role + state. Confirmation dialogs for destructive transitions.
- **Searchable Link dropdowns** — Link/Dynamic Link fields use a hybrid autocomplete: load first 50 records on focus, client-side filter, server search fallback for large datasets. Target doctype and display field read from config.
- **Back-referencing** — when viewing a document, the schema response includes `referenced_by` (all doctypes with Link fields pointing here). A "Related Documents" panel shows linked records.
- **Per-field validation errors** — unique constraints, required fields, and other validation errors display inline on the specific field with red border + error text.
- **Responsive** — sidebar collapses to hamburger menu on mobile. Forms and tables adapt to screen width.

### Config Versioning

```
Draft --activate--> Active --new version--> Superseded
                       ^                       |
                       +------ rollback -------+
```

Versions store full config snapshots + structured diff changelogs in `_kora_config_version`. Only one Active at a time. Schema changes are classified into Safety tiers (Safe/Warning/Blocked) via `schema.AnalyzeImpact()` before activation.

- `/workspace/auth/login` — login page (no sidebar, public)
- `/workspace` — dashboard with sidebar from `/api/system/navigation`

## Technology Stack

| Concern | Choice | Rationale |
|---|---|---|
| Language | Go 1.25 | Single binary, strong concurrency, fast startup |
| HTTP | Gin + `net/http` | Fast, well-documented, rich middleware ecosystem |
| Database | MySQL 8.0 | Well-understood DDL, INFORMATION_SCHEMA |
| Expressions | expr-lang/expr | Safe, sandboxed, fast |
| CLI | Cobra | Standard Go CLI framework |
| Admin UI | React 19 + TanStack + shadcn/ui + Tailwind | Config-driven SPA, embedded via go:embed |
| TLS | autocert (Let's Encrypt) | Free, automatic, zero-config |
| Security | gin-contrib/secure + cors | Production headers, CORS |
| Rate Limiting | x/time/rate | Token bucket, stdlib-adjacent |
| Config format | YAML (import), JSON (storage) | Human-readable + machine-friendly |
| AI / LLM | OpenAI GPT-4o, DeepSeek V4, Anthropic Claude | Multi-provider via OpenAI-compatible `/chat/completions` |
| AI Config | `AIConfig` struct + `_kora_secret` overrides | Per-model defaults, site-level customization |
| MCP | modelcontextprotocol/go-sdk | Stdio-based MCP server for Claude Desktop |
| Analytics | Go stdlib + MySQL | EventBus async CDC, daily/monthly rollup, query engine |
